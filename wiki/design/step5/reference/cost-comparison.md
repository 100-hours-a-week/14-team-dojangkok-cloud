## 4. 비교 요약

### V2 Prod (ASG 1~4) vs V3 (K8S + EC2 DB)

V2 기반 아키텍처는 유휴 시(ASG=1) 비용이 저렴하지만 이중화가 없고, 트래픽 피크 시(ASG=4) 서버가 4대까지 늘어나며 비용이 급증한다. K8S 기반의 V3는 트래픽과 무관하게 워커 노드 안에서 안전하게 파드를 스케일링한다.

| 구분 | V2 Prod (기본 ASG=1) | V2 Prod (피크 ASG=4) | V3 초기 (K8S Worker 3대) | V3 최종 (K8S Worker 6대, 3-AZ) |
|------|-----------------------|-----------------------|-------------------------|------------------------------|
| 컴퓨트 | ~$209 | ~$300 | ~$280 | ~$427 |
| 스토리지 | ~$24 | ~$24 | ~$37 | ~$45 |
| 네트워크 | ~$65 | ~$65 | ~$38 | ~$38 |
| **합계** | **~$298** | **~$389** | **~$355** | **~$510** |
| **비용 변동 (vs V2 기본)**| — | **+$91 (+30%)** | **+$57 (+19%)** | **+$212 (+71%)** |

> Phase 2(DB K8S 이관)가 폐기되어 EC2 DB 비용이 K8S 인프라 비용에 중첩되어 남았다.
> **V3 초기(Worker 3대)**: 컴퓨트 자원이 고정되어 V2의 피크(ASG=4, $389) 시점보다 오히려 **저렴하다(-$34/월)**. GCP 파편화 비용이 걷히며 K8S 오버헤드를 상당 부분 상쇄하여, 기본 대비 +19% 수준만으로 안정적인 자동 복구(Self-healing) 인프라를 달성한다.
> **V3 최종(Worker 6대)**: 트래픽이 본격적으로 성장하는 시점의 구성이다. 기본 비용은 높지만, Pod의 완전 분산(anti-affinity) 및 N-1 장애 내성 43%라는 막강한 고가용성을 보장한다.

## 5. 비용 변동 주요 요인

**절감 항목**
- **GCP 인프라 제거**: AI Server, ChromaDB를 K8S(AWS)로 통합 (-$127)
- **NLB 제거**: RabbitMQ 내부화 (-$18)
- **ASG 피크 비용 회피**: FE/BE가 VM 단위가 아닌 Pod 단위로 확장되므로 EC2 추가 스풀링 불필요.

**증가 항목**
- **K8S 인프라 추가**: CP(1대), Worker(3~6대), NAT(2→3대), EIP 등 (초기+$189, 최종+$336)
- **DB EC2 중복 비용**: DB를 K8S로 흡수하지 않아 기존 V2의 EC2 비용(~$76) 지속 발생.

## 6. 가치 비교

K8S 전환은 단순 비용 절감이 아닌, 서비스 신뢰도와 운영 효율을 교환하는 투자다.

| 가치 | V2 Prod (ASG=1) | V3 K8S Prod |
|------|---------|---------|
| 서비스 이중화 | 이중화 없음 (1대 장애 시 마비) | Pod 복제본 2 + anti-affinity |
| AZ 분산 | App 일부 2-AZ 배치 | **App 3-AZ 균등 분산** |
| App 장애 복구 | VM 복구 (수 분) | **Pod 즉각 재스케줄링 (수 초)** |
| DB 가용성 | 단일 인스턴스 (수동 복구) | K8S와 무관 (EC2 잔류, 원상태 유지) |
| 클라우드 분산 | AWS + GCP (운영 파편화, 기술 격차) | **AWS 100% 통합 (단일 운영/결제)** |
