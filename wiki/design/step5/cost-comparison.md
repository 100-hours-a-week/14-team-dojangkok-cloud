## 4. 비교 요약

### 동등 조건 비교

K8S는 Pod 복제본(FE×2, BE×2) + 3-AZ 분산 + N-1 장애 내성으로 서비스 이중화가 기본 포함.
V2에서 동등 수준은 ASG=2.

| | V2 Prod (ASG=2) | K8S Prod (A) | K8S Prod (B) |
|--|----------------|-------------|-------------|
| 컴퓨트 | $209 (AWS) + $111 (GCP) = $320 | $336 | $336 |
| 스토리지 | $24 | ~$42 | ~$57 |
| 네트워크 | ~$65 | ~$38 | ~$38 |
| **합계** | **~$334** | **~$416** | **~$431** |
| **차이** | — | **+$82 (+25%)** | **+$97 (+29%)** |

### V2 ASG=1 (현재 상태) 비교

| | V2 Prod (ASG=1) | K8S Prod (A) | K8S Prod (B) |
|--|----------------|-------------|-------------|
| **합계** | **~$298** | **~$416** | **~$431** |
| **차이** | — | **+$118 (+40%)** | **+$133 (+45%)** |

> 부하테스트 결과 BE/MySQL Request 상향 + 3-AZ N-1 요건으로 Worker 4→6대 증가가 비용 상승의 주 원인. V2 대비 비용은 증가하지만, **3-AZ 분산 + 자동 복구 + DB HA + 단일 클라우드 통합**이라는 운영 가치를 포함.

### 비용 변동 요인

**절감 항목**

| 요인 | 변동 | 금액 |
|------|------|------|
| GCP 인프라 전액 제거 (AI Server + ChromaDB + NAT/IP/GAR) | 감소 | -$127 |
| NLB 제거 (RabbitMQ 내부화) | 감소 | -$18 |
| V2 FE/BE/DB 인스턴스 대체 | 감소 | -$98 |
| **절감 소계** | | **-$243** |

**증가 항목**

| 요인 | 변동 | 금액 |
|------|------|------|
| Worker 6대 (t4g.large) | 증가 | +$294 |
| CP 노드 추가 (t4g.medium) | 증가 | +$30 |

## 5. 가치 비교

K8S는 V2 대비 **+$118~133/월 (+40~45%) 비용 증가**이지만, 이에 상응하는 운영 가치 포함:

| 가치 | V2 Prod (ASG=1) | K8S Prod |
|------|---------|---------|
| 서비스 이중화 | 이중화 없음 | Pod 복제본 2 + anti-affinity |
| AZ 분산 | 2-AZ (일부 서비스만) | **3-AZ 2-2-2 균등 배치** |
| 장애 내성 | 노드 장애 시 수동 복구 | **N-1 69%, AZ 장애 83%** |
| DB HA | 단일 인스턴스 | Primary-Replica (시나리오 B) |
| 장애 복구 | VM 재프로비저닝 (수 분) | Pod 재스케줄링 (수 초) |
| 단일 클라우드 | AWS + GCP 이원화 | AWS 통합 |
| 인스턴스 수 | 9대 (AWS 7 + GCP 2) | 10대 (CP 1 + Worker 6 + NAT 3) |
| 관리 도구 | AWS/GCP 콘솔 + Terraform | kubectl 단일 |

> **비용 정당성**: Worker 4→6대 증가는 부하테스트 결과(BE 700m, MySQL 500m Request 상향)와 N-1 장애 내성(69%) 확보를 위한 필수 선택. 4대로는 N-1 110%로 장애 시 Pod Pending 발생.
