# 도장콕 인프라 무중단 마이그레이션 전략 기술 문서 (V1 -> V2)

## 1. 서비스 개요 및 마이그레이션 배경

도장콕은 "AI와 함께 하는 똑똑한 집 구하기"를 슬로건으로 처음 집을 구하는 대학생, 사회초년생들이 주택 임대차 계약을 수행하는 과정에서 낯선 법률 용어와 복잡한 계약서 앞에서 겪는 불안감을 해소하고, 합리적인 의사결정을 할 수 있도록 도와주는 주택 임대차 계약 보조 서비스입니다.

### 무중단 마이그레이션을 준비하는 이유
마이그레이션은 단순한 코드 배포와 달리 서비스가 실행되는 인프라 전체의 뼈대를 갈아끼우는 거대한 작업입니다. 일반적으로 이런 대대적인 변경은 서비스 전체를 중단하는 점검 시간(Downtime)을 갖고 수행하는 것이 보통이지만, 도장콕은 이 인프라 교체 작업을 **무중단**으로 수행하기로 결정했습니다.

현재 V1 환경에서 도장콕이 제공하는 기능들은 사용자가 현저히 적은 새벽 시간대(02:00 ~ 05:00)에 잠깐의 다운타임을 가지더라도 비즈니스에 큰 타격이 없을 것이라 타협하기 쉽습니다. 하지만 도장콕이 지향하는 핵심 가치인 **"정보 격차 해소 및 주택 임대차 계약에 대한 주체적 이해 보장"**을 온전히 실현하기 위해서는, 다음의 핵심 기능들이 24시간 언제라도 단 1초의 끊김 없이 무중단으로 제공되어야 한다고 판단했습니다.

1. **쉬운 계약서 열람 (Read-Only):**
   쉬운 계약서 생성 요청은 긴 처리 시간으로 인한 Split-brain 발생을 막고자 마이그레이션 간 일시적으로 차단(503 안내)합니다. 하지만 기존에 분석 완료된 계약서와 위험 특약 조항을 다시 열람하는 기능은 유지되어야 합니다. 사용자가 계약서 초안이나 완료한 계약서를 가지고 쉬운 계약서를 새로 생성하는 작업보다, **큰돈이 오가는 계약을 앞두고 불안함에 기존 결과를 반복 조회하며 고민하는 빈도가 훨씬 높기 때문**입니다.
2. **집노트 통합 관리 (Read & Write):**
   현장에서 방을 보며 즉각 사진을 올리는 사용자도 있지만, 늦은 시간 사진첩에 모아둔 사진들을 조용히 한 번에 업로드하거나 저장된 여러 매물의 수압·채광 정보 등을 서로 비교하며 깊게 고민하는 경우도 매우 많습니다. **'사용자의 의사결정은 밤낮을 가리지 않는다'** 는 도장콕 팀의 철학에 따라, 주거 환경 선택에 필수적인 집노트의 모든 CRUD 기능은 심야 시간대에도 완벽히 무중단으로 제공되어야 합니다.

추가적으로 V2부터는 사용자 개인의 활동을 넘어 다수의 사용자가 복잡하게 상호작용하는 **'매물 커뮤니티'** 기능이 새롭게 추가됩니다. 이때부터는 다운타임 최소화가 서비스 신뢰도에 미치는 파급력이 비교할 수 없이 커집니다.
따라서 시스템 복잡도가 상대적으로 낮고 온전히 통제 가능한 현재의 V1 단일 인스턴스 시점에서 무중단 마이그레이션을 선제적으로 기획하고 리허설 수준으로 실행해 봄으로써, 향후 대규모 서비스를 운영해 나갈 **개발팀의 인프라 마이그레이션 숙련도를 끌어올리고 귀중한 운영 노하우를 축적**하고자 합니다.


### 절대 허용하지 않는 중단점 (Not Acceptable)
우리가 보호해야 할 코어 가치를 훼손하는 다음의 서비스 중단은 어떠한 경우에도 발생해서는 안 됩니다.
- **서비스 전체 통신 마비:** 전환 작업 중 단 1초라도 사이트(웹/앱) 자체가 다운되어 접속이 불가능해지는 전면적인 다운타임은 철저히 배제합니다.
- **데이터 무결성 훼손:** 사용자가 이미 작성해둔 매물 정보, 집노트 기록, 분석이 끝난 계약서 결과가 유실되거나 손상되는 것은 결정적인 신뢰 파괴로 직결됩니다.
- **AI 차단 장기 방치:** 일시적인 503 처리(부분 점검 제한)는 기술적 전환(Drain) 과정을 확보하기 위한 순간적 조치이므로, V2 마스터 전환이 끝나는 즉시 해제되어야 합니다.

### 점검 중 일시적으로 허용하는 중단점 (Acceptable)
- **핵심 원칙:** 서비스 전체가 완전히 멈추는 **전면적인 다운타임(Downtime)은 발생하지 않아야 합니다.**
- **기본 서비스 (일시적 허용 범위):** DB Master 전환(Failover 승격) 시점에 발생하는 **수 초 이내의 극히 일시적인 '쓰기(Write) 지연'만 허용**합니다. 
  - 집노트 사진 업로드 등 데이터 생성 요청 시 1~3초 정도 로딩(Spinner)이 길어지는 수준으로만 영향을 제한합니다.
  - 클라이언트(앱/웹) 수준에서 자동 재시도(Retry) 로직을 적용하여 사용자가 오류(Error)를 직접 마주하지 않도록 방어합니다.
  - 저장된 집노트나 기존 계약서 결과를 확인하는 **'읽기(Read)' 작업은 어떠한 순간에도 지연 없이 무중단으로 제공**되어야 합니다.
- **AI 연동 기능 (부분 점검 허용 범위 - Graceful Degradation):** '쉬운 계약서', '체크리스트' 등 **처리 시간이 약 5분 소요되는 동기식 AI 요청 기능은 마이그레이션 핵심 작업(Master 승격 준비 및 교체 시간) 기간 동안 일시적인 사용 제한(부분 점검, 예: 503 에러 응답)을 허용**합니다.
  - 평균 5분에 달하는 긴 HTTP 요청 대기 시간 도중에 DB가 Read-only로 전환될 경우 '쓰기 실패'나 데이터가 양쪽으로 분리되는 'Split-brain' 같은 치명적인 데이터 무결성 훼손이 일어날 수 있으므로, 이를 미연에 방지하기 위한 조치입니다.


## 2. 인프라 구성 현황 (AS-IS vs TO-BE)

### V1 인프라 (현재 - AS-IS)
<img src="https://raw.githubusercontent.com/100-hours-a-week/14-team-dojangkok-cloud/refs/heads/main/wiki/migrations/v2/images/v1-architecture.png"></image>
초기 도장콕 서비스의 시장 가능성을 확인하고 검증된 스펙이지만, 향후 폭발적인 트래픽 대응에는 한계를 가지는 뼈대입니다.
- **단일 서버 밀집 구조:** AWS 단일 인스턴스(EC2)에 Nginx, Next.js, Spring Boot, MySQL, Redis가 모두 패키징되어 배포됩니다. 
- **SPOF 관점의 위험 노출:** 단일 서버에 모든 의존성이 물려있어 단 하나의 컴포넌트에 장애가 생기거나 인스턴스를 재시작(Reboot)해야 할 때 서비스 전체 다운타임을 초래하는 심각한 위험을 내포하고 있습니다.
- **인입 및 제어 구조:** Route53 도메인이 단일 인스턴스의 Public IP로 직접 매핑되며, Nginx가 내부에서 TLS(HTTPS) 인증서를 직접 관리하므로 스케일링이 사실상 불가능합니다.

### V2 인프라 (목표 - TO-BE)
<img src="https://raw.githubusercontent.com/100-hours-a-week/14-team-dojangkok-cloud/refs/heads/main/wiki/migrations/v2/images/v2-architecture.png"></img>
추구하는 "폭넓은 연결 플랫폼(단기/양도 임대 등)"을 지탱하기 위해 고가용성(HA)과 확장성을 확보한 구조입니다.
- **분리형 스케일아웃 구조:** Next.js, Spring Boot, MySQL, Redis 그리고 비동기 메세징을 위한 **RabbitMQ**가 각각 독립된 머신(인스턴스)으로 분산됩니다.
- **고가용성 확보:** Next.js와 Spring Boot는 멀티 AZ(Availability Zone) 기반의 Auto Scaling Group(ASG)으로 묶어 트래픽 폭증에 유연하게 대응합니다.
- **부하 분산 및 트래픽 제어:** Route53 도메인을 ALB(Application Load Balancer)로 넘겨, 안정적인 트래픽 분배를 수행합니다.
- **보안 및 인증 관리:** AWS ACM을 활용하여 ALB 단에서 중앙 집중식으로 TLS 인증서를 관리합니다.


## 3. 무중단 마이그레이션 핵심 전략 및 절차

마이그레이션의 가장 큰 목표는 새벽 시간대(02:00 ~ 05:00)에 활발히 일어나는 사용자의 **집노트 사용 및 쉬운 계약서 열람 트래픽**이 끊기지 않고 V1 인프라에서 새로운 V2 인프라로 유실 없이 전환되도록 만드는 것입니다. 이를 위해 먼저 전체 마이그레이션 타임라인을 구성한 후 상세 전략을 따릅니다.

### ⏱️ 마이그레이션 타임라인 요약
마이그레이션에 영향을 주는 중추적인 포인트들을 통합한 시간순 요약입니다.
- **[사전 준비]** V2 인프라 완전 프로비저닝, DNS TTL 조절(60초 이하 최소화), VPC Peering 구축 완료
- **[AM 02:00]** V1(Master) ➡️ V2(Slave) 실시간 데이터 동기화 시작 (MySQL & Redis)
- **[AM 02:30]** 긴 처리 시간이 예상되는 AI 기능(쉬운 계약서 등) 일시 차단(503 안내) 및 In-flight 요청 드레인 대기 (약 5~10분)
- **[AM 02:40]** 완벽한 데이터 로드가 확인된 찰나의 순간, V1 DB Read-only 강제 락 전환 및 V2 단독 Master 전격 승격 (✔️ **수 초간 쓰기 지연**)
- **[AM 02:40]** 승격 직후 V1 Nginx 설정을 스위칭하여 전체 트래픽을 V2 ALB로 우회 및 AI 차단 해제 조치 적용 (✔️ **안전한 Nginx 우회망 확보**)
- **[AM 02:45]** Route53 DNS 타겟 레코드를 최종적으로 V2 ALB로 전환 (전파 지연 발생 시 V1 Nginx 우회망이 트래픽을 안전하게 방어)
- **[AM 03:00~]** 트래픽 동향 모니터링 및 잔여 V1 인스턴스 중지(Stop) 

### 상세 단계별 진행 가이드
### Step 1: V2 인프라 사전 구성 및 배포 (준비)
- V2 인프라(ALB, ASG, DB 개별 노드, RabbitMQ 등)를 완전히 프로비저닝합니다.
- 최신 버전의 도장콕 코드를 배포하고, 내부망에서 '쉬운 계약서' AI 연동 및 '집노트' 이미지 업로드 등 핵심 기능에 대한 기본 테스트를 마칩니다.

### Step 2: VPC 연동 및 실시간 데이터 동기화 (데이터 브릿지)
- 서로 다른 대역망에 있는 V1 VPC와 V2 VPC 간에 **VPC Peering**을 맺어 안전한 통신 터널을 뚫습니다.
- 유저 데이터(집노트, 가입자 정보, 분석 기록 등)의 유실을 방지하기 위해, **V1을 Master, V2를 Slave**로 세팅하여 MySQL과 Redis의 Replication(동기화)을 시작합니다.

### Step 3: 처리 시간이 긴 AI 기능 일시 차단 (부분 점검 시작, 승격 5분 전)
- **목적:** '쉬운 계약서', '체크리스트' 등 응답까지 약 5분이 소요되는 동기식 AI 요청이 마이그레이션 진행 도중 뒷북으로 처리되어 DB 쓰기 실패(Read-only 상태 제어) 및 Split-brain 현상을 유발하는 것을 사전에 방지합니다.
- **조치:** DB 스위칭 작업이 이루어지기 **최소 5분 전**에 선제적으로 V1 Nginx 설정을 업데이트하여 해당 AI 기능 API 엔드포인트에 대한 신규 요청을 차단하고 503(Service Unavailable) 메시지를 반환하도록 변경합니다. 이를 통해 현재 런타임에서 처리 중인 최장 5분짜리 기존 요청들이 안전하게 DB에 기록되고 완전히 비워질 때(Drain)까지 대기합니다.
- **효과:** 마이그레이션 진행 중 해당 기능만 점검 상태가 되며(우아한 기능 저하, Graceful Degradation), 사용자에게는 "안전한 업데이트를 위해 AI 기능만 일시 점검 중입니다"라는 안내를 통해 양해를 구합니다. 이외 기능(조회, 사진 업로드 등)은 5분 대기 시간에도 평소처럼 그대로 가동됩니다.

### Step 4: V1 DB Read-only 전환 및 V2 DB의 Master 승격 (스위칭)
- **목적:** V2 DB가 Master를 이어받는 찰나의 순간에 V1 DB로 예상치 못한 쓰기(Write)가 발생하여 양쪽 DB 간 데이터가 어긋나는 Split-brain 현상을 원천 차단합니다.
- **조치 1 (V1 쓰기 차단):** 데이터 동기화 지연율이 0에 수렴할 때, **가장 먼저 V1 DB(MySQL 등)를 강제로 Read-only 모드로 전환**합니다. 이 시점부터 V1 환경에서는 그 어떠한 신규 쓰기 트랜잭션도 발생하지 않습니다.
- **조치 2 (V2 마스터 승격):** V1에 더 이상 쓰기가 발생하지 않는 것을 확인한 직후 찰나의 순간에, 준비된 스크립트를 실행하여 완벽히 동기화된 V2 DB를 단독 Master로 승격시킵니다.

### Step 5: V1 Nginx 라우팅 프록시 업데이트 (트래픽 무중단 우회 및 AI 기능 복구)
- V2 DB가 Master로 전환됨과 동시에, V1 인스턴스의 Nginx 설정을 즉각적으로 교체(reload)합니다.
- **핵심 우회 및 복구:** 
  1. 기존 V1 Nginx로 인입되던 도장콕 사용자 전체 요청을 **V2 인프라의 ALB 주소로 Reverse Proxy** 하도록 변경하여, 새 서버 환경으로 안전하게 유도합니다.
  2. Step 3에서 예방을 위해 차단했던 **AI 기능 API의 차단을 해제**하여, 트래픽이 최신화된 V2 인프라 인스턴스에서 다시 응답을 줄 수 있도록 복구합니다.
- DNS가 전파되기 전 기존 IP로 유입된 잔여 트래픽 역시 이 조치를 통해 유실 없이 V2 환경으로 응답을 받게 됩니다.

### Step 6: Route53 DNS 라우팅 변경 (글로벌 전파)
- 도장콕 서비스 도메인(Route53)의 타겟 레코드를 V1의 단일 IP에서 **V2의 ALB 주소로 업데이트**합니다.
- 지역/통신사에 따라 발생하는 DNS 전파 지연(Propagation Delay) 시기에도, V1 IP로 접근한 구 사용자는 Step 5의 설정에 의해 안전하게 V2로 안내됩니다.

### Step 7: 트래픽 모니터링 및 V1 인프라 정리 (마무리 단계)
- DNS 전파가 완료되어가면서 V1 Nginx Access Log의 유입량이 서서히 0에 수렴하는지 모니터링합니다.
- 더 이상 V1으로 향하는 어떠한 사용자 요청도 찍히지 않는 것을 확인하면, 롤백을 대비하여 V1 인프라를 우선 **중지(Stop)** 상태로 둡니다.
- 충분한 모니터링(약 1~2주) 후, 집노트 및 계약 관련 모든 기능이 완전히 이상 없음을 확정 지으면 리소스를 최종적으로 완전 삭제(Terminate)합니다.


## 4. 비상 롤백 계획 (Rollback Plan)

전환 과정 중 어느 단계에서라도 의도치 않은 서비스 불안정이 감지될 경우, 신속히 V1 체제로 원상 복구하여 가용성을 최대한 지켜냅니다. V1 인프라를 삭제(Terminate)하지 않고 중지(Stop) 상태로 두는 이유도 이러한 롤백 경로를 확보하기 위함입니다.
- **AI 차단 적용 후 이상 감지:** V1 Nginx 설정만 원복하여 즉각 AI 기능을 재활성화합니다. 데이터 이동이 필요 없는 상태입니다.
- **V1 DB Read-Only 잠금 직후 이상 감지:** 즉시 V1을 쓰기 가능 상태(`super_read_only = 0, read_only = 0`)로 해제하여 V1 중심의 서비스를 계속 이어갑니다. 
- **Route53 DNS 변경(최종 단계) 후 이상 감지:** Route53 레코드를 다시 V1 IP로 복구하고, V1 Nginx의 ALB 우회 포워딩 설정을 원복하여 트래픽을 V1으로 다시 빨아들입니다. DNS 전파에 따른 미세한 혼선만으로 무결성 복원이 이뤄집니다.


## 5. 주요 리스크 및 사전 대응 시나리오

인프라 스위칭이라는 거대한 이벤트 속에서 예상되는 주요 엣지 케이스와 기술적 한계들을 선제적으로 방어합니다.

- **AI 요청 드레인(Drain) 도중 무리한 사용자 재시도:** 503 점검 응답 시 단순 백지가 아닌, 명확한 점검 시간 고지와 친절한 안내 메시지(Graceful Degradation)를 포함하여 지속적인 앱 클릭이나 새로고침에 의한 사용자 불만과 단기적인 서버 부하를 방지합니다.
- **In-flight 요청 미처리로 인한 드레인 실패:** 단순히 5분이라는 추정 시간에만 의존하지 않습니다. V1 Nginx의 Access Log(`tail -f`)를 지속 모니터링하여 해당 AI 엔드포인트의 잔여 트래픽이 모두 완전히 응답을 마치고 소멸되었는지 **육안으로 명확히 교차 검증**한 후 DB 락(Lock) 단계로 넘어갑니다.
- **마스터 전환 중 예상치 못한 강제 쓰기 발생 (MySQL Split-Brain 강력 방어):** V2 마스터(Master) 자격을 승격하기 직전, Nginx 트래픽 차단이라는 앱 수준의 추상적 제어에만 의존하지 않고 V1 MySQL 자체에 `super_read_only = 1` 락을 확립합니다. 이로써 서비스 백그라운드 프로세스나 기타 알 수 없는 경로가 유발할 수 있는 그 어떤 변칙적인 쓰기 접근도 DB 엔진 레벨에서 완벽히 차단합니다.
- **Redis 세션 강제 제어의 한계 보완:** MySQL과 달리 Redis는 Master 노드 자체의 쓰기를 전면 강제 차단하는 Native `read_only` 옵션이 존재하지 않습니다. 쓰기 차단을 오직 앱 앞단의 Nginx 트래픽 우회에만 의존해야 하므로, 스위칭 찰나의 세션 데이터 유실을 방지하기 위해 V2 자동 승격 전에 V1과 V2 Redis 간의 `master_repl_offset`이 100% 일치하는지 재차 엄격하게 검증합니다.
- **V2 ALB 헬스체크 실패로 인한 트래픽 블랙홀:** 전체 트래픽 방향타를 튼 직후 V2 ALB 뒤의 대상 그룹(Target Group)이 비정상일 경우 즉각적인 전면 다운타임을 부릅니다. V1의 Reverse Proxy 덮어쓰기나 Route53 DNS 스위칭을 단행하기 전에, 독립된 내부망 환경에서 V2 ALB 헬스체크 정상 통과와 HTTP 200 응답 여부를 반드시 체크하고 넘어가도록 합니다.
- **DNS 전파 지연 중 구 인프라로 접속하는 사용자 제어 (무중단 최종 방어선):** Route53 변경 이력이 전 세계 Name Server에 전파(Propagation)되기 전까지 기존 V1 IP로 유입되는 타임랙(Time-lag) 트래픽이 발생합니다. 마이그레이션 단계 5에서 미리 구축해둔 V1 Nginx 설정이 **이러한 모든 V1 인프라에 대한 트래픽을 V2 ALB로 자동 우회 포워딩(Reverse Proxy)** 하도록 강제하여, 사용자 시점에서는 0초 지연과 완벽한 데이터 정합성 만족을 이끌어 냅니다.
